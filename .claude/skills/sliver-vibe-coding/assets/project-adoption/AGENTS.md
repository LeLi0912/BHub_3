# AGENTS.md

任何时候使用中文，除非用户明确要求英文输出。

## 项目阶段

本仓库是半路接管、已有代码、已有资料或 AI 已经开发过的项目。AI 不能把它当空项目重来，也不能因为缺少 `dev-docs/` 就拒绝工作。必须先做只读接管审计，确认当前真实边界，再决定下一步。

## 接管规则

- 架构优先、设计优先、真源优先、严格验收。
- 编码前先读当前代码、README、docs、`dev-docs`、agent 指令文件、package/framework 文件、schema/migration、测试脚本、部署脚本和 Git 状态。
- 现有代码和文档可能过期，但都是证据，必须先审计再判断。
- 没有内部真源时，先建议建立真源索引；不能直接开始高风险功能开发。
- 如果 `dev-docs/` 已被非开发资料占用，不能强行改用途，要另选内部真源目录并让用户确认。
- 对已有框架、SDK、技术栈先评估：继续沿用、沿用但修正、局部替换、分阶段迁移或立即停止。
- 禁止补丁式修复、假成功、mock 冒充真实功能、无意义兼容和多套 owner 并存。
- 普通 bug、局部 UI、单文件改动可以窄修窄验；发现架构、数据、权限、部署、第三方或真源风险时再升级治理。

## 危险接管动作

以下动作必须先让用户确认：

- 创建或改写本文件、`CLAUDE.md`、Cursor rules 等 agent 宪法。
- 宣布 `dev-docs/` 或其他目录为内部唯一真源。
- 移动、归档、删除、批量重命名旧文档。
- 改产品名称、公开 README、外部 docs、UI 文案或营销定位。
- 删除旧 API、字段、路由、配置、缓存、兼容分支、生成合同或部署脚本。
- 初始化嵌套 `dev-docs` Git 仓库或改变提交/发布流程。
- 认为项目无法继续，只因为当前真源缺失。

## 接管顺序

```text
只读接管审计
  -> 项目阶段分类
  -> agent 宪法质量判断
  -> 内部真源索引
  -> 当前状态审计
  -> owner map / 架构边界
  -> 验收和停止条件
  -> 第一个安全任务
```

## Git 和隐私

- 提交前确认 git root、`git status --short`、ignored 文件、staged file list。
- 禁止 `git add .`。
- 如果项目中有第三方参考库、下载模板、旧 demo、真实客户资料、`.env`、密钥、数据库 dump、上传文件或内部 docs，必须先分类和忽略。
- 已经被 Git 跟踪的私密文件不能靠 `.gitignore` 解决，必须停下报告并让用户决定是否 untrack、拆分或清理历史。

## 交接规则

上下文过大、换窗口、交给另一个 agent 或阶段未闭合时，必须输出可复制交接文本：项目路径、分支、git 状态、最新提交、已改文件、验证命令、未验证项、漂移风险和下一步命令。
