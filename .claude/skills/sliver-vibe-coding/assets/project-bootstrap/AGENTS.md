# AGENTS.md

任何时候使用中文，除非用户明确要求英文输出。

## 项目阶段

本项目处于从零启动或空项目启动阶段。AI 不能直接写功能代码、直接选框架、直接搭页面，必须先把产品边界、第一闭环、技术路线、架构 owner、验收方式写入 `dev-docs/` 并得到用户确认。

## 核心规则

- 架构优先、设计优先、真源优先、严格验收。
- 先读 `dev-docs/README.md` 指向的当前真源，再做判断。
- 给非技术用户一个主推荐，不给多个平级方案让用户猜。
- 技术选型必须基于项目形态、第一闭环、平台能力、部署目标、第三方依赖、维护能力和商业约束。
- 前端必须先有设计方向、token 真源、组件复用规则和样式漂移防线。
- 后端必须先有责任边界、API 合同、数据 owner、权限/安全边界和验证方式。
- 共享业务语义只能有一个 owner。UI、controller、adapter 只做展示或映射，不能私自拥有业务规则。
- 禁止补丁式开发：不得用散落 if、临时兼容、硬编码、mock、假成功、sleep、全局 flag 绕过架构问题。
- 禁止无意义兼容。旧字段、旧接口、旧页面、旧产品概念会污染当前主线时，先让用户做取舍。
- 用户明确否定的概念必须从文档、代码、命名和计划里删除，不能换名字保留。
- 任何缺失证据都标记为 `未验证`，不能包装成已完成。

## 启动顺序

```text
产品边界
  -> 第一闭环
  -> 功能拆分和不做什么
  -> 技术路线
  -> 架构 owner map
  -> agent 宪法正式化
  -> 第一阶段实施真源
  -> 子阶段开发
  -> 验收和 Git checkpoint
```

## 必须先问清楚

- 谁会第一个使用这个项目。
- 第一版必须完成哪一个完整动作。
- 第一版明确不做什么。
- 是否涉及登录、角色、支付、上传、敏感数据、第三方服务、本地系统能力或生产环境。
- 用户能接受的上线方式、预算、维护方式和交付对象。

## Git 和隐私

- 编码前确认真实项目根目录，避免在下载目录、参考项目或混合资料目录初始化项目。
- 第一次提交前先审 `.gitignore`，保护 `.env`、密钥、数据库、本地上传、内部资料、真实案例、商业计划和第三方参考库。
- 如果 `dev-docs/` 是内部资料且不能推送，先确认双仓管理：代码仓忽略 `dev-docs/`，`dev-docs/` 可单独本地 Git。

## 停止条件

遇到以下情况必须停下并问用户：

- 技术栈、框架、数据库、权限、支付、第三方服务、部署方式要改变。
- 第一闭环、MVP、不做什么或商业边界不清楚。
- 缺少实施真源却准备开发标准或高风险功能。
- 当前证据和用户说法、旧文档、AI 记忆冲突。
- 验证需要真实密钥、生产数据、外部账号、远程部署或破坏性操作。
